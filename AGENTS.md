# Web Application Development Agents

Webアプリケーションの開発からリリースまでをカバーするエージェント群の定義と使い方。

---

## エージェント一覧

| エージェント | 役割 | 使用フェーズ |
|---|---|---|
| [planner](#planner) | 要件整理・アーキテクチャ設計・実装計画 | 計画フェーズ |
| [developer](#developer) | 機能実装・バグ修正 | 実装フェーズ |
| [tester](#tester) | テスト作成・実行・結果報告 | 検証フェーズ |
| [reviewer](#reviewer) | コードレビュー・セキュリティ確認 | レビューフェーズ |
| [deployer](#deployer) | ビルド・デプロイ | リリースフェーズ |
| [monitor](#monitor) | 本番監視・インシデント対応 | 運用フェーズ |

---

## 開発フロー

```
[要件・アイデア]
      │
      ▼
  planner ──── 実装計画、アーキテクチャ設計、タスク分割
      │
      ▼
  developer ── 機能実装、バグ修正
      │
      ▼
  tester ───── テスト作成・実行、失敗をdeveloperへフィードバック
      │ (テスト全通過)
      ▼
  reviewer ─── コードレビュー、セキュリティチェック
      │ (APPROVE)
      ▼
  deployer ─── staging → (確認) → production デプロイ
      │
      ▼
  monitor ──── 本番監視、インシデント対応
```

## 各エージェント詳細

### planner

**役割**: 要件分析からアーキテクチャ設計、実装計画の策定まで
**使用フェーズ**: 計画フェーズ（コードを書く前に必ず使う）
**使用タイミング**:
- 新機能の開発を始める前
- 新しいスプリントの開始時
- 技術的な方向性の決定が必要なとき

**できること**:
- 要件の整理とAcceptance Criteriaの定義
- 技術スタックの選定と根拠の提示
- アーキテクチャ設計（コンポーネント構成、データフロー）
- ファイル/ディレクトリ構成の提案
- ステップごとの実装計画の作成
- リスクと未解決の問いの洗い出し

**できないこと**: アプリケーションコードの作成

**呼び出し例**:
```
/planner ユーザー認証機能を追加したい。メール+パスワードとGoogle OAuthに対応。
```

---

### developer

**役割**: 計画に基づいた機能実装・バグ修正
**使用フェーズ**: 実装フェーズ
**使用タイミング**:
- plannerが計画を出した後
- テスターやレビュアーからバグ修正依頼が届いたとき

**できること**:
- 機能コードの実装
- バグの修正
- plannerの計画に従ったファイル作成・編集

**原則**:
- 必要最小限のコードのみ書く（過剰な抽象化・機能追加NG）
- セキュリティ脆弱性を絶対に混入させない
- テスト・レビューは他エージェントに委ねる

**呼び出し例**:
```
/developer plannerの計画に従って、ユーザー登録APIを実装してください。
```

---

### tester

**役割**: テストの作成・実行・失敗報告
**使用フェーズ**: 検証フェーズ（実装後、レビュー前）
**使用タイミング**:
- developerが実装を完了したとき
- バグ修正後の回帰確認

**できること**:
- ユニットテストの作成（関数単位）
- インテグレーションテストの作成（APIエンドポイント、DB接続）
- E2Eテストの作成（重要なユーザージャーニー）
- テストの実行と結果サマリーの報告

**失敗時の動作**: 自分でコードを修正せず、失敗内容と仮説をdeveloperに報告する

**呼び出し例**:
```
/tester 実装されたユーザー登録APIのテストを書いて実行してください。
```

---

### reviewer

**役割**: マージ前のコードレビュー・セキュリティ確認
**使用フェーズ**: レビューフェーズ（テスト通過後、デプロイ前）
**使用タイミング**:
- すべてのテストが通過した後
- mainブランチへのマージ前

**チェック内容**:
- セキュリティ（OWASP Top 10、シークレット漏洩等）
- 正確性（要件との一致、エッジケース）
- 保守性（可読性、不要コードなし）
- パフォーマンス（N+1クエリ、大量データ処理等）

**判定**:
- `APPROVE` → deployerへ進む
- `REQUEST CHANGES` → developerへ差し戻し
- `BLOCK` → セキュリティ問題。即時停止、ユーザーへ報告

**呼び出し例**:
```
/reviewer 実装されたユーザー認証コードをレビューしてください。
```

---

### deployer

**役割**: ビルド・staging/productionへのデプロイ
**使用フェーズ**: リリースフェーズ（reviewer承認後）
**使用タイミング**:
- reviewerがAPPROVEした後
- hotfixのリリース時

**デプロイ手順**:
1. pre-deployチェックリスト確認
2. ビルド実行・検証
3. stagingデプロイ（自動）
4. **productionデプロイ（ユーザー確認必須）**
5. ヘルスチェック
6. 問題があれば自動でロールバック提案

**重要**: productionへのデプロイは必ずユーザーの明示的な承認を得てから実行する

**呼び出し例**:
```
/deployer stagingにデプロイしてください。
/deployer productionへリリースします（reviewer承認済み）。
```

---

### monitor

**役割**: 本番監視・インシデントトリアージ・障害報告
**使用フェーズ**: 運用フェーズ（リリース後）
**使用タイミング**:
- productionデプロイ直後（30分間の集中監視）
- 定期的な健全性確認
- エラー・パフォーマンス劣化の調査

**インシデント重要度**:
- **P1** — 全ユーザーに影響（即時エスカレーション）
- **P2** — 主要機能が壊れている
- **P3** — 軽微な劣化

**できること**:
- エラーレート・レスポンスタイムの確認
- ログのスキャンと異常パターンの検出
- 根本原因の診断
- 修正推奨（実装はdeveloperへ委ねる）

**呼び出し例**:
```
/monitor デプロイ後の本番環境を30分間モニタリングしてください。
/monitor 昨夜からエラーが増えています。調査してください。
```

## よくあるフロー例

### 新機能開発（フルフロー）
```
1. /planner [機能の説明]
2. /developer [plannerの計画を参照して実装]
3. /tester [実装されたコードのテスト]
   → 失敗: /developer [テスト失敗内容を修正] → 3に戻る
4. /reviewer [テスト通過済みコードのレビュー]
   → REQUEST CHANGES: /developer [修正] → 3に戻る
5. /deployer staging にデプロイ
6. [動作確認] → /deployer production にデプロイ
7. /monitor デプロイ後監視
```

### バグ修正（緊急）
```
1. /monitor [バグの調査・根本原因特定]
2. /developer [monitoの報告を元に修正]
3. /tester [修正の確認テスト]
4. /reviewer [修正のレビュー]
5. /deployer [hotfixをproductionへ]
6. /monitor [修正確認]
```
